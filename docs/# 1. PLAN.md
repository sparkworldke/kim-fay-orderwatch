# Core Admin Connectors API + Env MVP

## Summary
Build the first functional admin connector slice: live Laravel APIs and a live React administration page for AI keys, Acumatica config/health, roles/permissions display, notification rules display/toggle, and audit logs. Add safe env defaults/examples for local API connectivity and required third-party credentials. Do not implement Outlook OAuth/sync, cron execution, or the full notification engine in this slice.

## Key Changes
- Backend services:
  - Add `EncryptionService`, `AuditLogger`, and `StructuredLogger`.
  - Add `AiConnectorService` for env/database key precedence, encryption, masking, status, and delete fallback.
  - Add a lightweight `AcumaticaService` for config storage, credential validation stub/HTTP-ready shape, health response, and masked credential handling.
- Backend controllers/routes:
  - Add `/api/admin/ai-keys`, `/api/admin/acumatica`, `/api/admin/acumatica/validate`, `/api/admin/roles`, `/api/admin/permissions`, `/api/admin/notification-rules`, `/api/admin/audit-logs`, `/api/admin/health`.
  - Guard routes with `auth:sanctum` and an admin middleware compatible with current `users.role === Administrator`; permission-table enforcement can be layered later.
  - Return only masked credential previews; never return raw keys/passwords/secrets.
- Frontend:
  - Replace static `src/routes/app.administration.tsx` content with TanStack Query-driven panels.
  - Add hooks/types/schemas for AI keys, Acumatica, roles, permissions, notification rules, audit logs, and health.
  - Functional MVP actions: save/delete AI provider key, update/validate Acumatica config, toggle notification rules.
  - Use existing shadcn/ui components, skeleton loading states, API error empty states, and toast feedback.
- Env files:
  - Add required non-secret variables to `backend/.env.example`: Acumatica, Microsoft OAuth, OpenAI, Anthropic.
  - Add frontend API base examples to root `.env.example`.
  - Fix local root `.env` API URL typo from `puplic` to `public`, using safe non-secret values only.
  - Do not commit real API secrets.

## Interfaces
- `GET /api/admin/ai-keys` returns provider rows for `openai` and `anthropic`: `{ provider, source, masked_preview, last_used_at, health_status }`.
- `POST /api/admin/ai-keys` accepts `{ provider: "openai"|"anthropic", key: string }`.
- `DELETE /api/admin/ai-keys/{id}` deletes stored DB key and falls back to env.
- `GET /api/admin/acumatica` returns masked config, health, and latest sync logs.
- `PUT /api/admin/acumatica` accepts config fields and encrypts credential fields.
- `POST /api/admin/acumatica/validate` returns `{ success, message, response_ms }`.
- `GET /api/admin/roles`, `GET /api/admin/permissions`, `GET /api/admin/notification-rules`, `PUT /api/admin/notification-rules/{id}`, `GET /api/admin/audit-logs`, and `GET /api/admin/health` provide the admin page data.

## Assumptions
- This slice prioritizes a working admin connector MVP over the entire Kiro module.
- Current `users.role` remains the source of admin access for now; role/permission tables are exposed and seeded but not yet the full authorization engine.
- Third-party secrets stay blank in examples and local files unless supplied later outside source control.
- Outlook OAuth, mailbox sync, cron job execution, full notification dispatch, and full demo-data migration are deferred to later implementation slices.
